## Anti-Patterns Verdict

**Does this look AI-generated? No — but the dashboard carries several systematic tells that undermine an otherwise clean design.**

The core workspace is intentional: dark-first, ember accent, Cormorant/Inter pairing, surface-layer elevation. No gradient text, no glassmorphism, no hero-metric templates. These are good.

The problems are structural and repeated:
1. The overview stat cards use `border-top: 3px solid` in five distinct colors (ember, blue `#8ba4ff`, yellow `#f5c842`, neutral, ember again) as a status taxonomy. This is the side-stripe anti-pattern applied to the top edge — same logic, same problem. It signals the design reached for the most obvious status differentiator (color stripe) rather than a considered system.
2. Three sidebar labels ("WORKSPACE", "ADMIN", "ACCOUNT") are the banned uppercase-tracked eyebrow pattern, replicated at every nav section break.
3. The readiness gate uses another uppercase eyebrow kicker inline.

## Detailed Findings by Severity

### P1 — Major

**[P1] Interactive element inside role="link" div**
- **Location**: `dashboard/page.tsx` ~line 959–1055 (project card render); `page.module.css:3957` `.projectMenuButton`
- **Category**: Accessibility
- **Impact**: ARIA spec forbids interactive descendants inside `role="link"`. Screen readers (NVDA, VoiceOver) handle this inconsistently — some will shadow the button, others will expose both. Keyboard focus order is also confusing: card focuses as a "link", then the three-dot button gets its own tab stop inside it.
- **WCAG**: 4.1.2 Name, Role, Value; ARIA in HTML §2.5 (interactive element descendants of interactive elements)
- **Recommendation**: Replace `<div role="link" tabIndex={0}>` with a proper layout: a wrapping `<article>` or `<div>`, a real `<a href="...">` covering the main click target, and the menu button positioned absolutely (or via CSS) outside the anchor's interactive area. This is the correct pattern for card + overflow-menu combos.
- **Suggested command**: `/impeccable harden dashboard project cards`

**[P1] No focus trap in modals**
- **Location**: All modals in `dashboard/page.tsx` — add-project, workspace articles, user workspace, connect WordPress, platform selection
- **Category**: Accessibility
- **Impact**: Modals declare `role="dialog" aria-modal="true"` but implement no focus containment. Keyboard users pressing Tab will cycle through the background page content while the modal is open. AT users relying on `aria-modal` to virtualize the dialog get inconsistent behavior across screen readers.
- **WCAG**: 2.1.2 No Keyboard Trap (Escape should dismiss), ARIA dialog pattern (focus must stay in dialog)
- **Recommendation**: Implement focus trap: on modal open, capture all focusable elements within the dialog, intercept Tab/Shift+Tab to cycle within that set, and restore focus to the trigger on close. Use a library (`focus-trap-react`, `@radix-ui/react-dialog`) or implement a `useFocusTrap` hook. Also set `focus()` to the first focusable element (or the dialog itself if `tabIndex="-1"`) on open.
- **Suggested command**: `/impeccable harden dashboard modals`

**[P1] Unlabeled admin table inputs**
- **Location**: `dashboard/page.tsx` ~line 1087–1147 (users admin table rows), `<input placeholder="Full name" />` and `<input ... value={u.subscription_type} />`
- **Category**: Accessibility
- **Impact**: Inputs inside `<td>` cells have no `<label>` element and no `aria-label`/`aria-labelledby`. Screen readers announce "edit text" with no contextual name. Users navigating by form controls cannot understand the field's purpose without reading surrounding cell content, which AT doesn't always associate.
- **WCAG**: 1.3.1 Info and Relationships; 4.1.2 Name, Role, Value
- **Recommendation**: Add `aria-label="Full name"` / `aria-label="Subscription type"` to each input, or use `aria-labelledby` pointing to the column header `<th>`. The `<th>` already exists in the table head; `scope="col"` and `id` on each header would let inputs reference them.
- **Suggested command**: `/impeccable harden dashboard admin table`

**[P1] Inverted `--aa-primary-active` — active state lightens instead of darkening**
- **Location**: `dashboard/dashboard.module.css` `.dashboardDark` block, line declaring `--aa-primary-active: color-mix(in oklab, #d97757, #ffffff 14%)`
- **Category**: Theming
- **Impact**: When a button is pressed (`:active`), the ember accent lightens to a pale orange. Pressed states should always darken (more pressure = more ink) to match physical affordance intuition. This is the inverse of what every UI system does and what DESIGN.md specifies (`--aa-primary-active: #c86a4c` — a darker ember). Any button using this token communicates the wrong interaction feedback.
- **WCAG**: 1.4.11 Non-Text Contrast (state changes must be discernible)
- **Recommendation**: Change to `color-mix(in oklab, #d97757, #000 14%)` or simply set `--aa-primary-active: #c86a4c` as documented in DESIGN.md.
- **Suggested command**: `/impeccable polish dashboard theming`

**[P1] Off-spec accent colors in status card taxonomy (top-border system)**
- **Location**: `dashboard/dashboard.module.css` — `.wsOverviewChartCard` (border-top: ember), `.wsOverviewPipelineCard` (border-top: #8ba4ff), `.wsOverviewPanel_scheduled` (border-top: #f5c842), `.wsOverviewPanel_draft` (border-top: #8ba4ff), `.wsOverviewPanel_pending` (border-top: rgba(255,255,255,0.45)), `.wsOverviewPanel_published` (border-top: ember)
- **Category**: Anti-Pattern + Theming
- **Impact**: Five different top-border colors introduce a three-color accent system (ember, blue, yellow) that breaks the one-accent rule. The colors `#8ba4ff` and `#f5c842` are hardcoded, untokenised, and not referenced anywhere in DESIGN.md. The top-border mechanic is the side-stripe anti-pattern applied to the top edge — a color stripe as the primary status differentiator rather than a considered semantic system. On its own this is weak, but across five cards it becomes a visual taxonomy built from an unintentional palette.
- **Recommendation**: Replace the status stripe system entirely. Options: (1) Status badge/pill with an icon (Published, Draft, Scheduled, Pending) using a defined semantic token set; (2) Subtle background tint per status (single-hue opacity ramps from the ember accent or neutral); (3) Icon-only status indicator in the card. If color coding is required, define status tokens (`--aa-status-scheduled`, `--aa-status-draft`) in globals.css with proper values, and document them in DESIGN.md.
- **Suggested command**: `/impeccable polish dashboard status cards`

### P2 — Minor

**[P2] `transition: width` on readiness progress fill**
- **Location**: `dashboard/dashboard.module.css:1041` `.overviewReadinessProgressFill { transition: width 0.35s ease }`
- **Category**: Performance
- **Impact**: `width` is a layout property. Animating it triggers layout + paint on every frame, preventing GPU compositing. On lower-end devices this can cause visible jank during the progress fill animation.
- **WCAG**: N/A (performance)
- **Recommendation**: Use `transform: scaleX()` with `transform-origin: left` instead of `width`. The fill element gets `width: 100%` and `transform: scaleX(0.73)` (the percentage). Parent needs `overflow: hidden`. The transition becomes `transition: transform 0.35s ease`.
- **Suggested command**: `/impeccable optimize dashboard`

**[P2] Workspace article modal without virtualization**
- **Location**: `dashboard/page.tsx` workspace modal render (articles table) — up to 1500 rows rendered as DOM nodes simultaneously
- **Category**: Performance
- **Impact**: At 1500 articles, the browser creates ~1500 `<tr>` elements with image thumbnails, text, badges, and buttons. Initial render will be slow; scrolling will drop frames; memory pressure rises. On mobile this can be severe enough to stall the tab.
- **Recommendation**: Virtualise the list using `@tanstack/react-virtual` or `react-window`. Render only the visible rows + a small overscan buffer. Alternatively, paginate the modal table (25 or 50 per page) — simpler, no library dependency, and arguably better UX for a modal context.
- **Suggested command**: `/impeccable optimize dashboard workspace modal`

**[P2] No lazy loading on workspace overview thumbnail images**
- **Location**: `dashboard/page.tsx` — `wsOverviewFeaturedImg` images in the workspace articles table
- **Category**: Performance
- **Impact**: Images below the fold download immediately on page load. For a workspace with many articles, this is unnecessary bandwidth and can delay LCP on the above-fold content.
- **Recommendation**: Add `loading="lazy"` to the `<img>` elements (or use Next.js `<Image>` with its built-in lazy loading). If using `<Image>`, set meaningful `width`/`height` to prevent layout shift.
- **Suggested command**: `/impeccable optimize dashboard`

**[P2] No `aria-live` on dynamic content regions**
- **Location**: `dashboard/page.tsx` — error states (`{error ? <div ...>{error}</div> : null}`), success/loading indicators, project list refresh
- **Category**: Accessibility
- **Impact**: When errors appear or the project list refreshes, screen readers receive no announcement. AT users only discover errors if they navigate back to that area of the page.
- **WCAG**: 4.1.3 Status Messages
- **Recommendation**: Wrap error zones in `<div role="status" aria-live="polite">` (polite) or `aria-live="assertive"` for critical errors. For the project refresh, add a visually-hidden `<div aria-live="polite">` that announces "Projects updated" after a successful reload.
- **Suggested command**: `/impeccable harden dashboard`

**[P2] Modals missing `aria-describedby`**
- **Location**: `dashboard/page.tsx` — all modals have `aria-label` but no `aria-describedby`
- **Category**: Accessibility
- **Impact**: When AT announces a dialog, it reads the accessible name (`aria-label`) but cannot point to the dialog's body text as a description. For modals with instructional content (e.g., the WordPress connect modal with plugin instructions), users miss the context announced on focus.
- **WCAG**: 1.3.1 Info and Relationships
- **Recommendation**: Add `id="dialog-desc-[name]"` to the first descriptive paragraph in each modal and reference it with `aria-describedby="dialog-desc-[name]"` on the dialog element.
- **Suggested command**: `/impeccable harden dashboard modals`

**[P2] Hard-coded platform colors in add-project modal**
- **Location**: `dashboard/page.tsx` — inline `style={{ color: '#21759b' }}` (WordPress blue), `style={{ color: '#5f7f1a' }}` (Shopify green)
- **Category**: Theming
- **Impact**: Injects two brand colors that have no representation in the design system. In dark mode these colors are untested against the modal background. They also cannot be updated globally and violate the one-accent rule during the critical first-run flow.
- **Recommendation**: Remove inline color. Platform logos already carry brand recognition; the name text doesn't need the brand color applied. Use `var(--aa-body-strong)` for the platform name. If platform color differentiation is needed, consider a small platform-color dot/swatch as a design-system-controlled pattern.
- **Suggested command**: `/impeccable polish dashboard add-project modal`

**[P2] Hard-coded `color: '#fff'` inline styles in admin section**
- **Location**: `dashboard/page.tsx` — inline `style={{ color: '#fff' }}` on plan section heading and elsewhere in admin tab
- **Category**: Theming
- **Impact**: Bypasses the token system. In a light-mode variant, this text would be invisible on a light background. Even in the current dark-first setup, it can't be updated globally and creates maintenance drift.
- **Recommendation**: Replace `style={{ color: '#fff' }}` with a CSS class using `color: var(--aa-body-strong)` or `color: var(--aa-ink)`. No inline styles for token values.
- **Suggested command**: `/impeccable polish dashboard theming`

**[P2] Undefined token `var(--aa-radius-panel, 14px)` in `.tableScroll`**
- **Location**: `dashboard/dashboard.module.css` — `.tableScroll { border-radius: var(--aa-radius-panel, 14px) }`
- **Category**: Theming
- **Impact**: `--aa-radius-panel` is not defined in globals.css. The fallback `14px` resolves correctly at runtime but this is a phantom token — change the value in one place and the other won't follow. It also indicates design system drift (new token invented in isolation).
- **Recommendation**: Replace with `var(--aa-r-lg)` (the documented large-radius token, currently 4px but should be 12px after the globals.css fix). Or define `--aa-radius-panel` explicitly in globals.css.
- **Suggested command**: `/impeccable polish dashboard theming`

**[P2] `window.confirm()` on deleteUser — browser native dialog**
- **Location**: `dashboard/page.tsx` — `deleteUser()` function calling `window.confirm("Delete user...")`
- **Category**: Accessibility + Anti-Pattern
- **Impact**: Browser `confirm()` blocks the main thread, cannot be styled, ignores the design system, and is not keyboard-navigable in the same way as a designed modal. Screen readers may not announce it consistently across browsers. Also flagged in the critique (P2).
- **Recommendation**: Replace with a designed confirmation modal: "Delete [name]? This cannot be undone." with "Cancel" (secondary) and "Delete user" (danger primary, e.g. red or desaturated error tone) buttons.
- **Suggested command**: `/impeccable harden dashboard delete user`

**[P2] `window.alert()` on WordPress plugin download error**
- **Location**: `dashboard/page.tsx` — WordPress connect modal, `window.alert()` on download failure
- **Category**: Accessibility + Anti-Pattern
- **Impact**: Same issues as `window.confirm()` — unstyled, inconsistent AT support, off-brand.
- **Recommendation**: Render an inline error state within the modal: `<div role="alert">` with the error message and a retry link/button.
- **Suggested command**: `/impeccable harden dashboard wp connect modal`

**[P2] `.projectMenuButton` touch target is 34×34px**
- **Location**: `page.module.css:3957` `.projectMenuButton { width: 34px; height: 34px }`
- **Category**: Responsive / Accessibility
- **Impact**: WCAG 2.5.5 (AAA) recommends 44×44px touch targets; WCAG 2.5.8 (AA, WCAG 2.2) requires at least 24×24px with no adjacent targets within 24px. At 34px this is below AAA guidance. On mobile, this is the primary action button for accessing the project context menu — a mis-tap here could activate the project card navigation instead.
- **Recommendation**: Increase to `width: 40px; height: 40px` minimum (44px preferred). The button already has `border-radius: 999px`; the visual appearance scales naturally.
- **Suggested command**: `/impeccable adapt dashboard`

**[P2] `.wsOverviewReadinessKicker` — uppercase tracked eyebrow in readiness gate**
- **Location**: `dashboard/dashboard.module.css` — `.overviewReadinessKicker` (uppercase tracked label above readiness bar)
- **Category**: Anti-Pattern
- **Impact**: Additional instance of the banned uppercase-tracked eyebrow pattern, separate from the sidebar titles. Creates a secondary occurrence of the same tell.
- **Recommendation**: Replace with a regular sentence-case label at the same font size, or use a numeric level indicator (e.g., "Ready · 73%") without all-caps tracking.
- **Suggested command**: `/impeccable polish dashboard`

### P3 — Polish

**[P3] Multiple `!important` declarations in dark-context chart tooltip**
- **Location**: `dashboard/dashboard.module.css` — `.dashboardDark .wsChartTooltip *` and related selectors
- **Category**: Theming
- **Impact**: The tooltip style block fights against the `.dashboardDark` link-color rule using `!important` overrides. This is a specificity debt that becomes harder to resolve as styles grow. Indicates the tooltip was styled after the fact rather than within the design system's cascade.
- **Recommendation**: Move tooltip token overrides into a `.wsChartTooltip` scoped block and raise specificity there (`.wsChartTooltip .wsChartTooltipValue`) rather than using `!important`. Or use CSS layers to isolate the tooltip's own cascade.
- **Suggested command**: `/impeccable polish dashboard tooltip`

**[P3] Tooltip z-index `10000` is arbitrary**
- **Location**: `dashboard/dashboard.module.css` — `.wsChartTooltip { z-index: 10000 }`
- **Category**: Anti-Pattern
- **Impact**: Arbitrary large z-index values like 10000 create maintenance nightmares — they become the floor that everything else races above. A tooltip should sit at a defined `--z-tooltip` level (typically 700–900 in a semantic z-index scale).
- **Recommendation**: Define a semantic z-index scale in globals.css: `--z-dropdown: 100; --z-sticky: 200; --z-modal-bg: 300; --z-modal: 400; --z-toast: 500; --z-tooltip: 600`. Replace `10000` with `var(--z-tooltip)`.
- **Suggested command**: `/impeccable polish dashboard`

**[P3] Local tooltip CSS variables disconnect from global token system**
- **Location**: `dashboard/dashboard.module.css` — `--tip-bg: #222228`, `--tip-ink`, `--tip-label`, `--tip-muted`, `--tip-border` declared locally
- **Category**: Theming
- **Impact**: The tooltip has its own mini design system that doesn't inherit from `--aa-*` tokens. In a light-mode variant, this tooltip would be dark regardless of theme. Minor now (dark-first), but creates coupling debt.
- **Recommendation**: Map tooltip local vars to global tokens where possible: `--tip-bg: color-mix(in oklab, var(--aa-canvas), #fff 4%)`, `--tip-ink: var(--aa-body-strong)`, `--tip-muted: var(--aa-muted)`. The border can stay custom if needed.
- **Suggested command**: `/impeccable polish dashboard tooltip`

**[P3] No `prefers-reduced-motion` on global animations used by dashboard**
- **Location**: `globals.css` — `@keyframes aaGlowFloat`, `aaSpin`, `aaBlink` have no `@media (prefers-reduced-motion: reduce)` guard
- **Category**: Accessibility
- **Impact**: If the dashboard loading overlay or spinner uses these animations, users who prefer reduced motion still see them. (The `dashboard.module.css` skeleton animations correctly guard against this — the gap is in globals.css.)
- **WCAG**: 2.3.3 Animation from Interactions (AAA), best-practice for WCAG AA intent
- **Recommendation**: Wrap each `@keyframes` usage in a `@media (prefers-reduced-motion: no-preference)` block, or add `@media (prefers-reduced-motion: reduce) { .aaGlowFloat, .aaSpin, .aaBlink { animation: none; } }` to globals.css.
- **Suggested command**: `/impeccable polish globals`

**[P3] 2182-line monolithic component — no code splitting for admin sections**
- **Location**: `dashboard/page.tsx` (entire file)
- **Category**: Performance
- **Impact**: Admin sections (users table, plan management, limits) are bundled for all users, even non-admins who never see them. At 2182 lines there is also no memoization of the admin table renders, meaning admin-only renders happen on every state change.
- **Recommendation**: Lazy-import admin tab content with `const AdminUsersTab = dynamic(() => import('./AdminUsersTab'), { ssr: false })`. Split admin sections into their own sub-components behind an `isAdmin` gate. Low priority since Next.js code-splits at the route level, and the page is server-side rendered — this matters most for the admin sections' React component tree size.
- **Suggested command**: `/impeccable optimize dashboard`

**[P3] `wsOverviewStatCard` decorative gradient**
- **Location**: `dashboard/dashboard.module.css` — `.wsOverviewStatCard { background: linear-gradient(165deg, rgba(255,255,255,0.04) 0%, rgba(255,255,255,0.01) 100%) }`
- **Category**: Anti-Pattern
- **Impact**: A gradient that goes from 4% opacity to 1% opacity is indistinguishable from a flat surface at normal viewing angles and display brightnesses. It adds CSS complexity without visible benefit. If any decorative gradient warrants existence, it should be visible enough to provide the intended hierarchy signal.
- **Recommendation**: Replace with `background: var(--aa-surface-card)` (flat, consistent with other surface tokens) or increase the gradient range to be intentionally visible (0% → 6–8%). Invisible decoration is complexity for no return.
- **Suggested command**: `/impeccable quieter dashboard stat cards`

## Patterns & Systemic Issues

**1. Hardcoded colors used as status indicators (5 occurrences)**
`#8ba4ff`, `#f5c842`, and `rgba(255,255,255,0.45)` appear as top-border status markers across five dashboard card types. These are not in the design system, not tokenised, and establish a de-facto second accent palette that conflicts with the one-accent rule. This needs a design-system decision: either define status semantic tokens or replace the top-border system entirely.

**2. Uppercase tracked labels used as nav section separators (3+ occurrences)**
"WORKSPACE", "ADMIN", "ACCOUNT" in the sidebar + readiness kicker in the overview panel. The pattern is being reached for as a default separator mechanic. Replace with a consistent alternative: thin `<hr>` dividers, icon+label rows, or collapsible section headings.

**3. Inline `style` props bypassing the token system (8+ occurrences)**
`style={{ color: '#fff' }}`, `style={{ fontSize: 13 }}`, `style={{ marginBottom: 12 }}`, `style={{ color: '#21759b' }}`, etc. throughout the page component. Inline styles can't be updated from a central token, can't respond to theme switches, and create noise when reading component code. All recurring values should be CSS classes; one-off layout nudges should at minimum use `calc(var(--aa-space-*))`.

**4. Modal accessibility gaps are consistent across all 5 modals**
Every modal in the dashboard shares the same pattern: correct `role`, `aria-modal`, and `aria-label`, but missing focus trap, missing `aria-describedby`, and no managed focus on open/close. This is a single systemic fix (a shared `<Modal>` wrapper component) rather than five separate fixes.

## Positive Findings

- **Dark-mode token override system** in `.dashboardDark` is well-structured — all `--aa-*` tokens are properly re-declared, giving the workspace a distinct visual context without per-element dark-mode rules.
- **Skeleton loading states** in `dashboard.module.css` correctly implement `@media (prefers-reduced-motion: reduce)` on the shimmer animation — the right pattern, applied consistently for the dashboard's own skeleton bones.
- **Visibility-aware interval refresh**: the 90-second project reload checks `document.visibilityState !== 'visible'` before fetching — no unnecessary background requests when the tab is hidden.
- **Modal dimensions**: `min(980px, calc(100vw - 32px))` for the workspace modal adapts gracefully across viewport widths.
- **Focus styles**: `.projectCard:focus-visible`, `.projectMenuButton:focus-visible`, and all sidebar links have explicit focus-visible styles — keyboard users have clear affordances on every interactive element that's properly an element.
- **Off-canvas sidebar**: mobile nav is implemented with a proper off-canvas pattern (hamburger, `mobileTopBar`, off-canvas overlay), not a responsive-collapse that hides content.
- **Responsive table strategy**: admin tables wrapped in `.tableScroll` containers with `overflow-x: auto` prevents horizontal page scroll on mobile.
- **ARIA on the menu**: the project kebab button has `aria-label="Project options"` (or similar) — correctly named even without a visible text label.
